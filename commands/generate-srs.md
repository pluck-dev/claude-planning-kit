# /generate-srs - 요구사항 명세서(SRS) 생성

고객과 합의할 **소프트웨어 요구사항 명세서(SRS)**를 인터랙티브하게 작성합니다.
프로젝트 착수 전 범위와 기능을 정의하는 문서입니다.

## 수행 절차

### Phase 1: 기존 자료 확인

1. 프로젝트에 이미 코드가 있는지 확인합니다:
   - 코드가 있으면 → 코드 분석 후 역공학으로 요구사항 추출
   - 코드가 없으면 → 순수 인터뷰 기반으로 작성

2. 기존 문서를 확인합니다:
   - `docs/`, `README.md`, 기획서, 화면설계서 등

### Phase 2: 고객 인터뷰 (AskUserQuestion 활용)

순서대로 질문하며 요구사항을 수집합니다:

**라운드 1 - 프로젝트 개요:**
- 프로젝트명은 무엇인가요?
- 이 시스템이 해결하려는 핵심 문제는?
- 주요 사용자(타겟)는 누구인가요? (예: 내부 직원, 일반 고객, 관리자)
- 예상 사용자 수 규모는? (10명 이하 / 100명 / 1000명+)

**라운드 2 - 기능 요구사항:**
- 시스템에 필요한 핵심 기능을 나열해주세요 (키워드로)
- 사용자 역할별로 어떤 기능이 필요한가요?
- 이미 비슷한 시스템을 사용 중인가요? (벤치마크)
- 반드시 포함해야 할 기능 vs 있으면 좋은 기능을 구분해주세요

**라운드 3 - 비기능 요구사항:**
- 기술 스택 선호가 있나요? (프레임워크, DB 등)
- 모바일 대응이 필요한가요? (반응형 / 앱 / 불필요)
- 외부 시스템 연동이 필요한가요? (결제, SMS, 이메일, 소셜 로그인 등)
- 성능/보안 요구사항이 있나요?

**라운드 4 - 일정/예산:**
- 프로젝트 예상 기간은?
- MVP(최소 기능 제품)와 풀버전을 나눌 건가요?
- 유지보수 계약이 포함되나요?

### Phase 3: SRS 문서 생성

`docs/srs/` 디렉토리에 문서를 생성합니다:

#### 문서 구조

```
docs/srs/
├── 00-표지.md              # 프로젝트명, 버전, 작성일, 작성자, 고객사
├── 01-개요.md              # 프로젝트 배경, 목적, 범위, 용어 정의
├── 02-이해관계자.md         # 사용자 역할, 페르소나, 사용 시나리오
├── 03-기능-요구사항.md      # FR-001~N 넘버링, 우선순위(필수/권장/선택)
├── 04-비기능-요구사항.md    # 성능, 보안, 호환성, 접근성
├── 05-시스템-아키텍처.md    # 기술 스택, 시스템 구성도, 배포 환경
├── 06-데이터-요구사항.md    # 핵심 데이터 엔티티, 관계도(ERD)
├── 07-외부-연동.md          # 외부 API, 결제, SMS 등
├── 08-화면-목록.md          # 페이지/화면별 기능 목록, 와이어프레임 참고
├── 09-일정-마일스톤.md      # Phase별 기능 범위, 예상 일정
├── 10-제약사항-리스크.md    # 알려진 제약, 리스크, 완화 방안
└── 11-부록.md              # 참고 자료, 변경 이력
```

#### 기능 요구사항 형식

각 기능은 다음 형식으로 작성합니다:

```markdown
### FR-001: [기능명]

| 항목 | 내용 |
|------|------|
| **ID** | FR-001 |
| **우선순위** | 필수 / 권장 / 선택 |
| **설명** | [기능 상세 설명] |
| **사용자** | [어떤 역할이 사용하는지] |
| **입력** | [필요한 입력 데이터] |
| **출력** | [기대 결과] |
| **비즈니스 규칙** | [적용되는 규칙/조건] |
| **검증 기준** | [이 기능이 완료되었다고 판단하는 기준] |
```

### Phase 4: 검토 요청

생성 완료 후 사용자에게:
1. 전체 문서 요약을 보여줍니다
2. 누락된 요구사항이 없는지 확인을 요청합니다
3. 우선순위 조정이 필요한 항목이 있는지 물어봅니다

## 출력 형식

- 모든 문서는 **한국어 마크다운**으로 작성합니다
- 시스템 구성도, 데이터 관계도는 **Mermaid 다이어그램**을 사용합니다
- 기능 요구사항은 **FR-001** 형식으로 넘버링합니다
- 비기능 요구사항은 **NFR-001** 형식으로 넘버링합니다

## 중요 원칙

- 고객이 기술적 배경이 없어도 이해할 수 있는 용어를 사용합니다
- 모호한 표현 대신 구체적인 수치/조건을 명시합니다 (예: "빠르게" → "3초 이내")
- 범위 밖(Out of Scope) 항목도 명시하여 추후 분쟁을 방지합니다
- 기존 코드가 있으면 반드시 코드를 분석하여 현재 구현 상태를 반영합니다
